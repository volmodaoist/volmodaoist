# 个人博客问题分析与优化

## v0.0.1

项目 v0.0.1 版本之中，作为一个静态的个人博客站点，使用了原生 JavaScript (TypeScript) 进行 DOM 操作和页面渲染。虽然数据量目前较小，页面加载和交互没有明显的卡顿，但从代码实现的角度来看，确实存在一些可以优化的 Anti-pattern，尤其是在频繁和直接操作 DOM 的地方，因为直接操作 DOM 会频繁触发浏览器的 **回流 (Reflow)** 和 **重绘 (Repaint)**，是一种低效的行为。本小节将对 `src/components` 和 `src/pages` 目录下的代码进行性能分析。

- 渲染笔记列表的 `renderNotesList()` 函数中，当前的实现是在循环中直接操作真实 DOM。如果一页有 $N$ 篇笔记，浏览器就会连续触发 DOM-Tree 重新计算和渲染的复杂度则为 $O(N)$，可以使用 `DocumentFragment` 批量插入节点， 作为内存中的“文档碎片”容器，循环结束之后，一次性将这个碎片添加到真实 DOM。

  ```javascript
  // src/pages/notes.ts
  paginatedNotes.forEach(note => {
      // ...
      const card = document.createElement('div');
      // ...
      listContainer.appendChild(card); // 每次循环都直接操作真实 DOM
  });
  ```

  

- 首页包含 `home`, `publications`, `awards`, `projects` 四个 section，没有一处会包含公式，但是每个 section 加载完成后都会触发 MathJax 渲染，相当于在短时间内对整个网页进行了四次繁重的数学公式解析与排版，完全是不必要的。直接移除即可。

  ```typescript
  // src/pages/home.ts
  SECTION_NAMES.forEach(async (name) => {
      // ... 异步加载 markdown 并转化为 html
      element.innerHTML = html;

      // MathJax 公式渲染
      if (typeof MathJax !== 'undefined' && MathJax.typeset) {
          MathJax.typeset(); // 全局渲染被调用了四次, 实际只有笔记详情页才会涉及公式
      }
  });
  ```

- 函数 `applyConfig()` 通过遍历对象逐个更新 DOM，典型的在循环中直接修改 DOM 的行为，虽然目前 v0.0.1 配置项不多，但是如果配置项变多，依然会触发多次重排。

  ```typescript
  // src/components/Navbar.ts
  Object.keys(config).forEach(key => {
      const el = document.getElementById(key);
      if (el) {
          el.innerHTML = config[key]; // 循环触发 DOM 插入
      }
  });
  ```

## v0.0.2

### 1. 编辑体验优化方案

这个版本发现了一个严重影响编辑体验的问题：修改了静态 Markdown 文件之后，刷新主页内容竟然没有任何变化。

- **静态资源缓存问题与强制刷新方案**：分析发现，我们使用了原生 `fetch` API 来获取 `.md` 文件，浏览器会对这类静态资源执行强缓存策略，当资源 URL 没有变化，浏览器会直接从本地磁盘缓存中读取数据，而不是向服务器发起新的请求。

  - **[方案1: 拼接时间戳]** 最初的想法是在 URL 后面加上当前的时间戳作为 Cache-Busting，虽然这个方法可确保每次获取最新内容，但是 **默认禁用所有缓存**会加重网络请求的负担，浪费服务器带宽，影响页面加载速度，对于一个小规模博客来说可能无关紧要，但这并非一个优雅的做法。
    ```typescript
    const timestamp = new Date().getTime();
    const response = await fetch(`${CONTENT_INCLUDES_DIR}${name}.md?t=${timestamp}`);
    ```

  - **[方案2: 手动刷新按钮]** 为了兼顾性能和灵活性，最终决定保留默认缓存，但在页脚增加一个相对隐蔽的 "Refresh Cache" 按钮，手动点击拉取新数据，具体实现上面，通过存储一个新时间戳作为 `v` 参数，强制请求新文件；随后哪怕是正常的 F5 刷新也会带着 `v` 请求，从而持久化保持最新内容。
    ```typescript
    const renderMarkdown = async (forceRefresh: boolean = false) => {
        if (forceRefresh) {
            localStorage.setItem('content_version', new Date().getTime().toString());
        }
        const version = localStorage.getItem('content_version');
        const cacheBuster = version ? `?v=${version}` : '';
    
        // ...
        const response = await fetch(`${CONTENT_INCLUDES_DIR}${name}.md${cacheBuster}`);
    };
    ```



### 2. 刷新回退现象分析
然而，按照手动刷新按钮方案实现之后，点击 "Refresh Cache" 紧接着刷新，页面竟然又变回了陈旧的缓存状态。为此，我们通过  HTTP 缓存控制指令 `cache: 'reload'` 强制更新。这种“刷新回退”现象的原因在于，个别现代浏览器或 HTTP 缓存代理服务器，尤其是静态托管服务，
- 对于形如 `?v=xxx` 格式的特定 query 参数依然可能执行缓存。
- 在短时间内进行刷新，浏览器甚至可能复用底层的 Http 连接，导致数据未更新。

```typescript
const renderMarkdown = async (forceRefresh: boolean = false) => {
    // 1. 如果是强制刷新操作，我们需要记录当前的时间戳作为最新的"版本号"
    if (forceRefresh) {
        const currentTimestamp = new Date().getTime().toString();
        localStorage.setItem('content_version', currentTimestamp);
    }
    
    // 2. 获取本地存储中的版本号，以决定我们拉取哪个版本的文件
    const version = localStorage.getItem('content_version');

    SECTION_NAMES.forEach(async (name) => {
        try {
            // 3. 构建请求的 URL，如果存在版本号则附加在最后
            let requestUrl = `${CONTENT_INCLUDES_DIR}${name}.md`;
            if (version) {
                requestUrl = `${requestUrl}?v=${version}`;
            }

            // 4. 构建 fetch 请求的选项
            let fetchOptions: RequestInit = {};
            if (forceRefresh) {
                // 当用户明确要求强制刷新时，我们需要给浏览器下达不使用缓存的指令
                // reload 会强制浏览器重新发起网络请求，即使缓存中已存在
                fetchOptions.cache = 'reload';
            }

            // 5. 发起请求并渲染
            const response = await fetch(requestUrl, fetchOptions);
            // ...
        } catch (error) {
            console.error(error);
        }
    });
};
```
