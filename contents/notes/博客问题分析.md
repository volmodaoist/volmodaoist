# 背景

## v0.0.1

项目 v0.0.1 版本之中，作为一个静态的个人博客站点，使用了原生 JavaScript (TypeScript) 进行 DOM 操作和页面渲染。虽然数据量目前较小，页面加载和交互没有明显的卡顿，但从代码实现的角度来看，确实存在一些可以优化的 Anti-pattern，尤其是在频繁和直接操作 DOM 的地方，因为直接操作 DOM 会频繁触发浏览器的 **回流 (Reflow)** 和 **重绘 (Repaint)**，是一种低效的行为。本小节将对 `src/components` 和 `src/pages` 目录下的代码进行性能分析。

- 渲染笔记列表的 `renderNotesList()` 函数中，当前的实现是在循环中直接操作真实 DOM。如果一页有 $N$ 篇笔记，浏览器就会连续触发 DOM-Tree 重新计算和渲染的复杂度则为 $O(N)$，可以使用 `DocumentFragment` 批量插入节点， 作为内存中的“文档碎片”容器，循环结束之后，一次性将这个碎片添加到真实 DOM。

  ```javascript
  // src/pages/notes.ts
  paginatedNotes.forEach(note => {
      // ...
      const card = document.createElement('div');
      // ...
      listContainer.appendChild(card); // 每次循环都直接操作真实 DOM
  });
  ```

  

- 首页包含 `home`, `publications`, `awards`, `projects` 四个 section，没有一处会包含公式，但是每个 section 加载完成后都会触发 MathJax 渲染，相当于在短时间内对整个网页进行了四次繁重的数学公式解析与排版，完全是不必要的。直接移除即可。

  ```typescript
  // src/pages/home.ts
  SECTION_NAMES.forEach(async (name) => {
      // ... 异步加载 markdown 并转化为 html
      element.innerHTML = html;

      // MathJax 公式渲染
      if (typeof MathJax !== 'undefined' && MathJax.typeset) {
          MathJax.typeset(); // 全局渲染被调用了四次, 实际只有笔记详情页才会涉及公式
      }
  });
  ```

- 函数 `applyConfig()` 通过遍历对象逐个更新 DOM，典型的在循环中直接修改 DOM 的行为，虽然目前 v0.0.1 配置项不多，但是如果配置项变多，依然会触发多次重排。

  ```typescript
  // src/components/Navbar.ts
  Object.keys(config).forEach(key => {
      const el = document.getElementById(key);
      if (el) {
          el.innerHTML = config[key]; // 循环触发 DOM 插入
      }
  });
  ```
